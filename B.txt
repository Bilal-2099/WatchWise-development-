Given everything I know about your background, I'd actually **avoid** making an AI project where the AI is the main feature.

Why?

Your goal is to get hired as a **backend engineer**, not an ML engineer.

You want recruiters to look at your GitHub and think:

> "This person can design APIs, databases, authentication, background jobs, caching, deployment, and integrations."

The AI part should be a feature, not the product.

## My recommendation for you

Based on the projects you've already completed (Django REST APIs, chat app with Channels and Redis, and now FastAPI), I'd spend the next few months building **one substantial backend application** instead of several small ones.

A strong project should include:

* FastAPI
* PostgreSQL
* SQLAlchemy
* Alembic
* JWT authentication
* Redis
* Background tasks
* File uploads
* Docker
* Unit/integration tests
* Deployment
* One external API integration
* One AI-powered feature (summarization or recommendations)

That combination demonstrates the backend engineering skills most junior Python/FastAPI roles are looking for, while the AI component shows you can integrate modern services without making the project entirely dependent on them.

I actually think the Movie & Series project is the better choice.

The YouTube idea has several practical challenges:

* Large watch histories.
* Quota limits on the YouTube Data API.
* Privacy considerations.
* Long processing times.
* More expensive AI usage if you summarize many videos.

A movie and TV recommendation platform lets you demonstrate backend engineering without those complications.

## One suggestion that could make it stand out

Don't call it "Movie Recommendation System."

Build it as a **personal entertainment tracker**.

Imagine using apps like Letterboxd or MyAnimeList as inspiration, but with your own focus.

Features like:

* "Continue Watching"
* "Watched This Year"
* "Top Rated Movies"
* "Hidden Gems"
* "Weekend Picks"
* "Because You Liked *The Dark Knight*"

make it feel like a real product instead of a college project.

I also suggest planning it in milestones rather than trying to build everything at once:

1. **Core backend:** Authentication, movies, watchlist, ratings, reviews.
2. **Production features:** Pagination, filtering, search, Docker, tests, deployment.
3. **Advanced backend:** Redis, background jobs, recommendation generation.
4. **AI enhancements:** Personalized explanations and recommendations based on the user's viewing history.

That approach lets you have a deployable, interview-ready project early, while continuing to add advanced features over time.


We are building **CineCompass** (or **ReelMatch**), an industry-grade, personal entertainment tracker and streaming aggregator API using **FastAPI**, **PostgreSQL (SQLAlchemy 2.0)**, and **Redis**. Rather than bloating our database by replicating a massive film catalog, we are implementing a lean hybrid architecture: our PostgreSQL database will strictly manage user accounts and actions across four core tables (**Users, Watchlists, Ratings, and Reviews**), while all real-time movie data, deep search capabilities, and dynamic "Where to Watch" streaming links will be fetched on the fly from the **TMDB API**. We will develop this system in distinct, production-ready milestones—scaffolding the core relational database models and secure JWT authentication first, transitioning into query proxy endpoints, and ultimately scaling the application by introducing **Celery background workers** for advanced recommendation logic and offline tasks.

Phase 2 Design: The AI "CineProfile" Feature
We are refining the application’s AI feature by designing a dedicated GET /users/me/ai-persona endpoint that generates a sharp, witty, 3-to-4 sentence cinematic character summary based on the user's taste. Rather than wasting budget and slowing down performance by passing the user's entire watch history to the LLM, we are implementing an optimized filtering strategy: our PostgreSQL query will isolate only the user's top 5 to 10 highest-rated movies using a simple descending sort and limit constraint. This high "signal-to-noise" data bundle will be processed via a tight system prompt using a cost-efficient model (like gpt-4o-mini), resulting in an ultra-fast, budget-friendly feature that can be displayed proudly at the top of their user profile.
